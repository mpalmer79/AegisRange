# EVENT_SCHEMA.md

## Project Title
AegisRange: Adversary Emulation and Defensive Control Validation Platform

---

## 1. Purpose

This document defines the canonical event schema for AegisRange.

The event schema is the foundation of the system. All detection, response, and incident logic depends on:

- consistent event structure
- reliable metadata
- traceable relationships between events

The schema is designed for:
- observability
- correlation
- auditability
- deterministic detection

---

## 2. Design Principles

### 2.1 Canonical Structure
All events must conform to a single normalized structure regardless of source.

### 2.2 Immutability
Events are append-only and must never be modified after creation.

### 2.3 Traceability
Every event must support:
- correlation across requests
- linkage to actors and targets
- inclusion in incident timelines

### 2.4 Minimal Required Fields
Core fields are required for all events. Optional fields are allowed but must not replace required structure.

### 2.5 Context Preservation
Events must capture enough context to:
- reconstruct behavior
- explain detections
- support incident review

---

## 3. Event Categories

All events belong to one of the following categories:

- authentication
- session
- document
- system
- detection
- response
- incident
- scenario

---

## 4. Core Event Schema

Every event must include the following fields:

### 4.1 Identification
- event_id: unique identifier (UUID)
- event_type: specific action identifier (string)
- category: high-level grouping (string)

### 4.2 Timing
- timestamp: event creation time (ISO 8601)
- ingestion_timestamp: time event was stored

### 4.3 Actor Context
- actor_id: unique identifier of initiating entity
- actor_type: user, service, system
- actor_role: privilege level or role (optional)

### 4.4 Target Context
- target_type: resource type (document, session, etc.)
- target_id: identifier of target resource (optional)

### 4.5 Request Context
- request_id: unique request identifier
- correlation_id: links related events across a workflow
- session_id: session identifier (if applicable)

### 4.6 Source Context
- source_ip: originating IP address
- user_agent: client descriptor
- origin: logical origin (web, api, internal)

### 4.7 Outcome
- status: success or failure
- status_code: optional numeric or symbolic code
- error_message: optional error detail

### 4.8 Risk Metadata
- severity: informational, low, medium, high, critical
- confidence: numeric or categorical confidence score
- risk_score: optional computed value

### 4.9 Payload
- payload: structured JSON object containing event-specific data

## 5. Event Type Definitions

### 5.1 Authentication Events
Examples:
- authentication.login.success
- authentication.login.failure

Payload fields:
- username
- authentication_method
- failure_reason (if applicable)

### 5.2 Session Events
Examples:
- session.token.issued
- session.token.revoked

Payload fields:
- token_id
- session_state
- authentication_strength

### 5.3 Document Events
Examples:
- document.read.success
- document.download.success

Payload fields:
- document_id
- classification
- sensitivity_score

### 5.4 Detection Events
Examples:
- detection.rule.triggered

Payload fields:
- rule_id
- rule_name
- matched_conditions
- contributing_event_ids

### 5.5 Response Events
Examples:
- response.step_up_auth.enforced
- response.session_revocation.executed
- response.download_restriction.applied

Payload fields:
- action_type
- target_id
- reason
- enforcement_scope

### 5.6 Incident Events
Examples:
- incident.created
- incident.updated

Payload fields:
- incident_id
- incident_status
- linked_alert_ids

### 5.7 Scenario Events
Examples:
- scenario.step.executed
- scenario.run.completed

Payload fields:
- scenario_id
- step_name
- step_index

---

## 6. Correlation Model

### 6.1 Correlation ID
All events generated within a single logical flow must share a correlation_id.

Example:
- login attempt
- session issuance
- document access
- detection trigger

All linked through one correlation_id.

### 6.2 Request ID
Each API request generates a unique request_id.

Multiple request_ids may map to a single correlation_id.

### 6.3 Incident Linkage
Events must support linking to incidents through:
- incident_id (added post-detection)
- contributing_event_ids in detection events

---

## 7. Event Lifecycle

1. Event is generated by a service
2. Event is normalized to canonical schema
3. Event is validated
4. Event is persisted
5. Event is evaluated by detection engine
6. Event may trigger alert or response
7. Event may be linked to incident

---

## 8. Validation Rules

All events must:
- include required fields
- use valid timestamp formats
- include actor context
- include correlation_id
- conform to category definitions

Invalid events must be rejected or flagged.

---

## 9. Storage Considerations

Events are stored in a structured format with:
- indexed fields for:
  - event_type
  - actor_id
  - correlation_id
  - timestamp

- support for:
  - time-range queries
  - incident reconstruction
  - detection evaluation

## 11. Anti-Patterns

The system must avoid:
- inconsistent field naming
- missing correlation identifiers
- embedding critical data only in payload
- mutable event records
- unstructured log-only events

## 13. Summary

The event schema defines the foundation of AegisRange.

All system capabilities depend on:
